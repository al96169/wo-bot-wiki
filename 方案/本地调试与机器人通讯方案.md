# 本地调试与机器人通讯方案

## 涉及项目
- wo-bot-control 机器人服务
- wo-bot-web-debug web调试页面

## 名词解释
- **机器人端**：Jetson 机器人设备（部署 wo-bot-control 提供服务，IP: 192.168.1.47）
- **控制端**：电脑 / 平板 / 手机（浏览器访问 web-debug 页面）
- **中继服务器**：在控制端电脑上运行的 Vite 开发服务器（IP: 192.168.1.53，端口 9093）

## 整体架构

```
[浏览器 (wo-bot-web-debug)]              [Vite 中继 (9093)]              [机器人 (wo-bot-control)]
      |                                        |                                |
      |-- 信令 WebSocket（经过中继代理）-------->|--→ 转发至机器人 :8765 --------->|
      |                                        |                                |
      |-- mDNS 发现（HTTP API）---------------->|--→ bonjour 扫描局域网 ---------->|
      |                                        |                                |
      |========== WebRTC 媒体 + DataChannel (P2P 直连，不经过中继) ===============>|
      |                                        |                                |
```

## 1. mDNS 服务发现

### 实现方式
- **机器人端**：wo-bot-control 使用 Python `zeroconf` 库注册 `_wobot._tcp.local.` 服务，广播到局域网
- **中继服务器**：Vite 开发服务器通过 `bonjour-service` 监听 `_wobot._tcp.local.`，提供 `/api/discover` HTTP 接口
- **Web 前端**：调用 `/api/discover` 获取设备列表，同时对常见同网段 IP（.47、.100-.103）做 WebSocket 快速探活作为补充

### 各平台支持情况

| 平台 | 原生 mDNS 支持 | 发现方式 |
|------|:---:|------|
| macOS | ✅ | mDNS API → PTR 查询即可发现 |
| Windows | ❌ | 通过中继服务器的 `/api/discover` HTTP 接口间接发现 |
| Android | ❌ | 同上，通过中继服务器 |
| iOS | ✅ | Safari 支持 mDNS，可直连或通过中继 |
| Flutter App | ✅ | 使用 `multicast_dns` 包原生查询 |

## 2. WebRTC 信令与媒体传输

### 信令通道（必须，无法省略）
WebRTC 协议自身**要求**在建立 P2P 连接前通过信令通道交换 SDP（会话描述）和 ICE 候选地址。当前实现：

- **WebSocket 作为信令通道**：浏览器 ↔ Vite 中继 ↔ 机器人
- 信令消息类型：`webrtc_offer`、`webrtc_answer`、`webrtc_ice_candidate`
- WebSocket 在 DataChannel 就绪前也承载业务消息，DataChannel 就绪后自动切换

### 媒体传输（P2P 直连）
视频流通过 WebRTC 在浏览器和机器人之间**直接点对点传输（UDP）**，不经过中继服务器：

```
机器人 (192.168.1.47)  ←──UDP/RTP──→  浏览器 (192.168.1.xxx)
```

- 编码格式：VP8（软编码）
- 双摄像头支持：cam0 (USB) + cam1 (CSI) 各占一个 Video Transceiver
- 延迟控制：wall-clock 时间帧丢弃，避免编码积压导致延迟递增

### DataChannel 业务通道
- 建立 WebRTC 连接时自动创建 DataChannel（label: `wobot-control`）
- **发送优先级**：DataChannel 就绪时走 DataChannel → 未就绪时降级到 WebSocket
- 承载：运动控制指令、云台指令、摄像头开关、系统查询等

### ICE 连通性与 mDNS .local 问题

浏览器生成的 ICE 候选常包含 `.local` mDNS 主机名（如 `xxx.local:12345`），在不同操作系统间可能不可解析。解决方案：

1. **Vite 中继传递真实 IP**：在 WebSocket 代理握手时，从 `x-forwarded-for` / `remoteAddress` 提取客户端真实 IP，作为 `client_ip` 参数传给机器人
2. **双候选策略**：机器人收到 `.local` 候选时，同时添加真实 IP 候选（高优先级 2122252543）和原始 `.local` 候选，优先尝试真实 IP
3. **aioice 猴子补丁**：强制选择特定 ICE 候选对，绕过 Python 3.7 下 STUN 连通性检查的限制

## 3. 中继服务器

### 角色定位
Vite 开发服务器（端口 9093，监听 0.0.0.0）承担三种职责：

| 功能 | 实现方式 | 说明 |
|------|---------|------|
| 静态资源服务 | Vite dev server | 提供前端页面（Vue SPA） |
| mDNS 发现中继 | `/api/discover` HTTP 接口 | 在 Node.js 层做 bonjour 扫描 |
| WebSocket 代理 | `/api/device-ws` WebSocket 升级 | 转发信令 + 注入 `client_ip` |

### 多设备访问
中继服务器在控制端 A（如 Mac）启动后，同一局域网下的其他设备（Windows、平板）均可通过 `http://<A的IP>:9093/` 访问，无需各自启动中继。

## 4. 重连与保活

- **WebSocket 断开重连**：指数退避重连（首次连接 vs 保活断开采用不同策略），最大重试次数可配置
- **WebRTC 媒体超时检测**：连接建立后 5 秒内未收到视频数据，自动 `resetAndOffer()` 重建连接（最多 3 次）
- **ICE 状态监控**：`iceConnectionState` 进入 `completed` 后启动 5 秒兜底定时器，防止 `connectionState` 不变化导致假死
- **前端 WebRTC 监控面板**：实时显示 ICE/DataChannel/Signaling 状态和候选信息，便于调试

## 5. 约束规则

- **严禁机器人通过 RESTful API 提供服务** —— 所有交互通过 WebSocket 信令 + WebRTC DataChannel
- **机器人仅暴露端口 8765（WebSocket）** —— 无需开放 HTTP/REST 端口，减少攻击面
- **WebSocket 是信令和业务兜底通道，不是"最后手段"** —— WebRTC 协议要求必须有信令通道，WebSocket 是业内标准选择
- **媒体和数据走 P2P 直连** —— 中继服务器只做信令转发和 mDNS 发现，不解包/不缓存媒体流
